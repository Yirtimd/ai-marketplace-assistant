# PROJECT_RULES.md

# Правила разработки проекта AI Marketplace Assistant

Этот файл содержит основные правила разработки проекта.
Все изменения в кодовой базе должны соответствовать архитектуре, описанной в `ARCHITECTURE.md`, `ROADMAP.md` и `AGENTS.md`.

Главная цель правил — сохранить **чистую, масштабируемую и предсказуемую архитектуру**.

---

# 1. Общие принципы разработки

Проект строится как **модульная AI-платформа**, а не как монолитный скрипт.

Основные принципы:

* модульная архитектура
* разделение ответственности
* stateless агенты
* серверная память
* event-driven архитектура
* workflow orchestration

Каждый компонент системы должен иметь **четкую роль**.

---

# 2. Разделение слоев системы

Система делится на несколько слоев.

## Client Layer

Технологии:

React
Electron

Назначение:

* пользовательский интерфейс
* отображение данных
* отправка запросов к backend

Client **не содержит бизнес-логики**.

---

## Backend Layer

Технологии:

Python
FastAPI

Назначение:

* API
* управление данными
* запуск workflow
* управление задачами
* взаимодействие с внешними API

---

## AI Workflow Layer

Технология:

LangGraph

Назначение:

* управление сложными AI процессами
* запуск цепочек агентов
* принятие решений

---

# 3. Структура проекта

Структура backend должна строго соблюдаться.

```id="kpxr2v"
backend/

api/
orchestrator/
agents/
workflows/
services/
tasks/
database/
config/
```

Назначение папок:

api — HTTP endpoints
orchestrator — запуск workflow
agents — логика AI агентов
workflows — LangGraph workflow
services — работа с внешними API
tasks — фоновые задачи
database — модели и работа с БД
config — настройки приложения

---

# 4. Принципы разработки агентов

Агенты являются ключевым элементом системы.

Каждый агент должен:

* выполнять одну доменную задачу
* быть stateless
* не хранить состояние
* не вызывать напрямую других агентов

Агенты должны взаимодействовать **только через workflow**.

---

## Пример структуры агента

```id="52rm0k"
class AnalyticsAgent:

    def analyze_sales(self, sales_data):
        pass
```

Агент должен быть максимально простым.

---

# 5. Workflow правила

Все сложные AI процессы должны реализовываться через workflow.

Workflow реализуются через LangGraph.

Каждый workflow должен:

* иметь четкую цель
* иметь ограниченное количество шагов
* быть понятным и читаемым

---

## Пример workflow

```id="an6l9g"
load_sales_data
↓
analyze_sales
↓
decision_node
↓
recommend_action
```

---

# 6. Работа с состоянием

Состояние системы не хранится внутри агентов.

Состояние должно храниться:

PostgreSQL
Redis

LangGraph state используется только **внутри workflow**.

---

# 7. Работа с базой данных

Основная база данных:

PostgreSQL

База должна содержать основные сущности:

users
shops
products
sales
reviews
tasks
events
workflow_state

Все операции с базой должны выполняться через слой `database`.

---

# 8. Работа с внешними API

Все внешние API должны использоваться через `services`.

Примеры сервисов:

```id="vpkpbl"
services/wb_service.py
services/ai_service.py
services/image_service.py
```

Запрещено вызывать API напрямую из:

agents
workflows
api endpoints

---

# 9. Система задач

Фоновые задачи реализуются через Celery.

Назначение задач:

* периодические проверки
* долгие операции
* автоматизация процессов

Примеры задач:

* анализ продаж
* проверка остатков
* мониторинг цен

---

# 10. Event-driven архитектура

Система должна реагировать на события.

Примеры событий:

new_review
stock_low
sales_drop
price_change

События запускают workflow.

---

# 11. Принципы написания кода

Код должен быть:

* читаемым
* модульным
* хорошо структурированным
* легко расширяемым

Не допускается:

* огромные файлы
* смешивание логики
* дублирование кода

---

# 12. Добавление новой функциональности

При добавлении новой функции необходимо:

1. определить агент
2. определить workflow
3. определить события
4. определить задачи

Новая функциональность должна **вписываться в архитектуру**.

---

# 13. Работа с AI

AI используется только для задач, где это действительно необходимо.

Примеры:

* анализ текста
* генерация контента
* принятие решений

AI **не должен заменять обычную программную логику**.

---

# 14. Масштабируемость системы

Архитектура должна позволять:

* добавлять новых агентов
* добавлять новые workflow
* добавлять новые маркетплейсы

Система должна оставаться **модульной и расширяемой**.

---

# 15. Главная цель проекта

Создать систему:

AI Marketplace Manager

которая может:

* анализировать бизнес магазина
* принимать решения
* автоматизировать управление маркетплейсами

AI должен стать **операционным менеджером магазина**.
