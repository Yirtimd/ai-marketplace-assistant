# DEVELOPMENT_WORKFLOW.md

# Процесс разработки AI Marketplace Assistant

Этот файл описывает правила разработки и добавления новых функций в проект.

Цель документа — обеспечить последовательную разработку и сохранить архитектуру системы.

Все изменения должны соответствовать файлам:

ARCHITECTURE.md
PROJECT_RULES.md
AGENTS.md
ROADMAP.md

---

# 1. Общий процесс разработки

Каждая новая функция должна проходить следующие этапы:

1. анализ задачи
2. определение агента
3. определение workflow
4. реализация backend логики
5. добавление API
6. добавление интерфейса

Нельзя сразу писать код без понимания архитектуры.

---

# 2. Добавление новой функции

При добавлении новой функции необходимо определить:

какой агент будет использоваться
какой workflow будет запускаться
какие данные нужны системе
какие действия должны выполняться

Пример.

Новая функция:

"AI анализирует падение продаж"

Процесс разработки:

1 определить analytics_agent
2 создать sales_analysis_workflow
3 определить источники данных
4 реализовать анализ
5 вернуть результат пользователю

---

# 3. Разработка агентов

Агенты должны быть простыми.

Каждый агент выполняет **одну задачу**.

Нельзя создавать универсальных агентов.

Пример правильного агента:

```id="u63o0l"
class InventoryAgent:

    def predict_stock_out(self, sales_data, stock):
        pass
```

---

# 4. Разработка workflow

Workflow реализуются через LangGraph.

Каждый workflow должен быть:

понятным
ограниченным по сложности
разбитым на шаги

Пример workflow:

```id="k4i4or"
load_data
↓
analyze
↓
decision
↓
action
```

---

# 5. Работа с базой данных

Все операции с базой должны проходить через слой database.

Нельзя:

обращаться к базе напрямую из workflow
обращаться к базе напрямую из API

Правильная схема:

```id="qecvfp"
API
↓
Service
↓
Database
```

---

# 6. Работа с внешними API

Все внешние API должны использоваться через сервисы.

Пример:

```id="c8axu7"
services/wb_service.py
```

API нельзя вызывать из:

agents
workflow
api endpoints

---

# 7. Работа с AI

AI используется только там, где это действительно необходимо.

Примеры:

анализ текста
генерация описаний
генерация изображений

AI нельзя использовать для:

простых вычислений
бизнес-логики
работы с базой данных

---

# 8. Добавление нового агента

Если системе требуется новая область логики:

создается новый агент.

Пример:

advertising_agent
supplier_agent
trend_agent

Шаги:

1 создать файл агента
2 описать функции агента
3 добавить workflow
4 подключить orchestrator

---

# 9. Работа с orchestrator

Orchestrator отвечает за запуск workflow.

Он должен:

определять тип запроса
выбирать workflow
запускать процесс

Пример:

```id="rgn2zc"
user request → workflow
event → workflow
scheduled task → workflow
```

---

# 10. Разработка API

Все HTTP endpoints должны находиться в папке:

```id="nghept"
api/
```

API должно:

получать запрос
вызывать сервис
возвращать результат

API не должно содержать сложной логики.

---

# 11. Фоновые задачи

Фоновые задачи используются для:

анализа данных
мониторинга
долгих операций

Используется Celery.

Примеры задач:

анализ продаж
проверка остатков
мониторинг цен

---

# 12. Добавление UI

UI добавляется после реализации backend.

Последовательность:

1 реализовать backend
2 создать API
3 реализовать интерфейс

Нельзя начинать разработку UI до backend.

---

# 13. Проверка архитектуры

Перед добавлением новой функции необходимо проверить:

не нарушает ли она архитектуру
можно ли реализовать ее через существующих агентов
нужно ли создавать новый workflow

---

# 14. Документирование

Каждая новая функция должна сопровождаться:

обновлением AGENTS.md
обновлением ROADMAP.md (если необходимо)

Это помогает сохранить контекст проекта.

---

# 15. Главный принцип разработки

Каждая часть системы должна быть:

простая
модульная
расширяемая

Система должна оставаться **понятной и управляемой даже при росте проекта**.
